## 1. システム全体像

### 1-1. ブロック図（入力 → 処理 → 出力）


```
【入力】                       【Arduino UNO R3】                     【出力】
メンブレンキーパッド（A/B/C/D） ───→┌──────────────────────────────┐──→ LED赤
                                  │ ・状態管理                    │──→ LED黄
                                  │ ・パターン生成（乱数）         │──→ LED青
                                  │ ・入力との正誤判定             │──→ LED緑
                                  │ ・ラウンド/スコア管理          │──→ LCD1602（ラウンド／ハイスコア表示）
                                  └──────────────────────────────┘──→ パッシブブザー（任意機能）
```

---

### 1-2. 状態遷移図（システムの「ふるまい」）

**↓ 自分の状態遷移を書いてください**
```
[電源ON]
  ↓ 初期化完了

[待機中(TITLE)] ──── #キー押下 ────→ [パターン提示中(SHOW)] ──── 表示完了 ────→ [入力待ち(INPUT)]
     ↑                                        ↑                                     │
     │ #キー押下                               │ パターン最後まで正解                  │ 誤入力 / (任意)時間超過
     │ (ラウンド長を3にリセット)                │ ラウンド長 +1                        │
     │                                        └─────────────────────────────────────┤
     │                                                                              ↓
     └──────────────────────────────── [ゲームオーバー(OVER)] ←──────────────────────┘
```


| 状態名 | この状態のとき、何をしているか |
|:--|:--|
| 待機中 (TITLE) | LCDに「Press # to START」を表示し、全LED消灯。#キー押下を待つ。押下時にラウンド長=3で初期化してSHOWへ遷移。 |
| パターン提示中 (SHOW) | 現ラウンド長分のランダムLEDパターンを生成し、順番に点灯/消灯して提示。キー入力は完全に無視。表示完了でINPUTへ遷移。 |
| 入力待ち (INPUT) | A/B/C/Dキー入力を1手ごとに受け付けて即時判定。一致なら次の手へ（自己ループ）。最終手まで正解なら次ラウンドとしてSHOWへ。不一致または時間超過でOVERへ。 |
| ゲームオーバー (OVER) | LCDに「Game Over / Round: X / Best: Y」を表示。不正解音をブザー出力。#キー押下を待ち、押下時にラウンド長=3へリセットしてTITLEへ戻る。 |

【補足】
- INPUT状態は、押されたキーが、その時点で期待している色と一致しているかを1回ずつ判定する
- 入力が正解 かつ まだ残りの入力がある : INPUT状態のまま、次のキー入力を待つ
- 入力が正解 かつ 現在ラウンドの最後まで入力完了 : → ラウンド数を+1し、SHOW状態へ戻る
- 不正解: → OVER状態へ移る
- SHOW状態でのキー入力は完全に無視
- #キーの役割: TITLE/OVERではSTART/RESTARTに使用。INPUT状態で # を押された場合、 INPUT状態の # は未定義キーとして無効入力扱いとし、入力待ちを継続する（ゲームオーバーにはしない）
- OVER状態は #キー以外を押しても反応しない
- OVER状態で # を押すと TITLE へ戻り、再度 # でゲーム開始となる

### 3-3. タイミング設計（delay vs millis）

> ※ 複数の処理を同時に動かす場合、`delay()`を使うと他の処理が止まります。  
> 各タイマーが `delay` か `millis` かを事前に決めておきます。

| 決定事項 | 値・方針 | 根拠 |
| --- | --- | --- |
| **pattern[] 配列の上限** | 最大 20 手 | 人間が現実的に記憶できる上限として設定。これ以上は array overflow の防止に使う |
| **LED 点灯時間 / 消灯時間** | 点灯 500ms / 消灯 150ms （仮置き、実装時に調整可） | 視認性確保の最低ライン。3-3 のタイミング設計に追記 |
| **デバウンス時間** | 50ms | 4章の記述を数値で補強 |
| **キーリピート抑制** | あり （押しっぱなし = 1回のみ受理） | 長押しで連打判定になるのを防ぐ。4章に追記 |
| **LED 排他制御** | 1色のみ点灯 （同時点灯なし） | ゲームルール上、複数LED が同時に光ると誤読しやすい |
| **gameState の値** | TITLE=0 / SHOW=1 / INPUT=2 / OVER=3 | 実装時の #define または enum の基準として明示 |

| 処理 | 周期 | delay / millis | 理由 |
|:--|:--|:--|:--|
| タイトル画面のキー監視（showTitle） | 10msごと | **millis** | #キー待ち中も処理を止めず、即応性を確保するため |
| キーパッドスキャン（scanKeypad） | 5〜10msごと | **millis** | 入力取りこぼしを防ぎ、1手ごとの即時判定を行うため |
| デバウンス判定 | 30ms | **millis** | チャタリングによる誤入力を抑えるため |
| パターン提示: LED点灯時間（showPattern） | 400ms/手 | **delay** | SHOW状態ではキー入力を無視する仕様で、単純な順次表示を優先するため |
| パターン提示: 消灯間隔（showPattern） | 200ms/手 | **delay** | 点灯の区切りを明確にして視認性を上げるため |
| 入力タイムアウト判定（任意） | 3000ms/手 | **millis** | 時間超過を監視しつつ他処理を止めないため |
| LCD更新（updateLCD） | 状態遷移時のみ | **millis** | 毎ループ更新を避け、表示ちらつきと無駄な処理を減らすため |
| ブザー鳴動（playSound, 任意） | 正解100ms / 不正解500ms | **millis** | 音を鳴らしながら状態更新・表示更新を継続できるようにするため |

## 4. 異常系・エラー処理の方針

| 異常ケース | 検知方法 | 対応方針 |
|:--|:--|:--|
| チャタリング（キーパッド） | 前回の有効入力から 50ms 以内に次の入力を検出した場合 | millis() で経過時間を管理。50ms 未満の入力は無効として破棄 |
| 同時押し（複数キー） | 6ピンスキャン中に複数の Row が同時に LOW になる場合 | その読み取りを無効として破棄。直前の状態を維持する |
| SHOW 状態でのキー入力 | gameState == SHOW のとき | スキャンは行うが結果を無視。状態遷移を発生させない |
| INPUT 状態での未定義キー（# など） | scanKeypad() の戻り値が A/B/C/D 以外 | 無効入力として破棄。誤入力扱いにはせず入力待ちを継続 |
| OVER 状態での # 以外のキー | gameState == OVER かつ '#' 以外を検出 | 無視。# キーのみ TITLE 遷移のトリガとして受け付ける |
| SHOW→INPUT 遷移時のキー持ち越し  | SHOW終了直後に押下が残っている可能性がある場合 | 遷移時に scanKeypad() を1回空読みして入力バッファをクリアする |
| 同時押し検出時のスキャン処理      | 6ピンスキャンで複数Rowが同時にLOWになった場合  | そのスキャンサイクル全体を無効とし、次のサイクルまで待つ（1サイクル無効化）|

## 6. AIレビュー記録（基本設計版）

### Q1: 構造の妥当性確認
> 「この基本設計書をもとに Arduino UNO R3 でコードを書くとき、  
> ピン不足・ライブラリの競合・状態設計の漏れはありますか？」

**AIの回答（要約）：**

- **結論**: UNO R3 の現行構成で、致命的なピン不足と明確なライブラリ競合はない。
- **ピン設計**: D2〜D12 と A0〜A5 の割当で成立。D0/D1 未使用も適切。4×4キーパッドを A/B/C/D/# に限定した 6 ピンスキャン方針は妥当。
- **競合リスク（将来拡張時）**: A4/A5 をキーパッド列に使用しているため I2C 追加時は再配線が必要。D10〜D12 を LCD で使用しているため SPI 追加時も競合の可能性あり。
- **状態設計の要修正点**: 「INPUT で # を誤入力扱い」と「INPUT で # などを無効入力として継続」が文書内で不一致。どちらかに仕様統一が必要。
- **実装前の明確化推奨**: OVER から # 押下時の遷移（TITLE 経由で再開するか）と、SHOW→INPUT 遷移直後のキー持ち越し防止（入力バッファ初期化）を詳細設計で明記すると実装が安定する。

**対応した内容：**

- 1-2 の補足に「INPUT中の # は誤入力（ゲームオーバー）」と記述していたが、要件定義書 3-4「未定義値は無効入力として扱い判定に使わない」と矛盾していた。4章の記述（無効入力として破棄・継続）が正しいため、1-2 の補足を修正した。
- OVER → TITLE → SHOW の遷移で # を2回押す運用は意図的な設計。現状維持とし、1-2 の補足にその旨を追記した。
- 4章の異常系に「遷移時に scanKeypad() を1回空読みしてバッファをクリアする」を追記した。
- 4章の同時押し対応の記述を「そのスキャンサイクル全体を無効とし、次のサイクルまで待つ」とより具体的に修正した。

### Q2: 詳細設計への引き継ぎ確認
> 「この基本設計書から詳細設計に進むとき、  
> まだ決まっていない・詳細化が必要な部分はどこですか？」

**AIの回答（要約）：**

- **定数の確定が必要**: pattern配列の最大長、LED点灯/消灯時間、入力タイムアウト、デバウンス時間、難易度上昇ルール（任意）を数値で固定する。
- **状態遷移の処理順を明文化**: 各状態（TITLE/SHOW/INPUT/OVER）で「毎ループ処理」「遷移条件」「遷移時の初期化」を擬似コードレベルで定義する。
- **scanKeypad() の詳細化が必要**: 6ピンスキャン手順（列駆動→行読取）、同時押し判定条件、1サイクル無効化、押しっぱなし時の1回入力化（エッジ検出）を明記する。
- **入力判定仕様の確定**: INPUTでA/B/C/D以外を受けた時の扱い、タイムアウト時の遷移、SHOW→INPUT直後の空読みバッファクリア手順を具体化する。
- **表示/音の実装方針を確定**: LCD更新タイミング（状態遷移時のみ等）、文字列フォーマット、正解音/不正解音の周波数・長さ・ノンブロッキング制御を決める。
- **乱数仕様を確定**: randomSeedの入力源、デバッグ時の固定シード有無、配列上限到達時の動作（打ち止め/周回/ゲームクリア）を定義する。
- **テスト基準を追加**: 単体（generatePattern/scanKeypad/waitInput）と結合（全状態遷移、誤入力、同時押し、再スタート）ごとの合格条件を事前に設定する。

**対応した内容：**

- 詳細設計への引継ぎに必要な主要定数を 3-3 に追記した。
- その他は詳細設計書で対応する。
