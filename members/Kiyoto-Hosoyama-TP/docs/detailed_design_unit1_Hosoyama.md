### `setup()` — 初期化処理

```
【処理の流れ】
1. ピンモードを設定する
   - PIN_BUTTON → INPUT_PULLUP
   - PIN_BUZZER → OUTPUT
   - （LCDはI2Cなのでピンモード設定不要）

2. シリアル通信を開始する（デバッグ用）
   - Serial.begin(9600)

3. LCDライブラリの初期化
   - LiquidCrystal_I2C lcd(0x27, 16, 2) を生成
   - lcd.begin(16, 2)

4. 変数・ゲーム状態の初期化
   - currentState = 0（待機中）
   - score, playerLane, obstaclePos[], scrollSpeed, isGameOver などを初期値に

5. 起動音（任意）
   - tone(PIN_BUZZER, 1000, 200) などで起動ジングル

6. LCDに「スタート待ち」など初期画面を表示
```

---

### `loop()` — メインループ

```
【処理の流れ（状態ごとに分岐）】

毎回やること：
- 入力を読む（readInputs()を呼ぶ）
- 現在時刻を取得する（millis()）

currentState が 0（待機中）のとき：
- センサー値を監視する（readSensor()）
- 検知条件を満たしたら → currentState = 1 に変更

currentState が 1（動作中）のとき：
- メイン処理を行う（doMainProcess()）
- 終了条件を満たしたら → currentState = 2 に変更

currentState が 2（完了）のとき：
- 完了表示をする（showResult()）
- リセット入力を待つ → ボタン押したら currentState = 0 に戻す

currentState が 3（エラー）のとき：
- エラー表示をする（showError()）
- リセット入力を待つ
```

**↓ 自分の loop() を書いてください**

```
【処理の流れ】

毎回やること：
- now = millis() を取得する
- pressed = readButton() でボタン入力を読む（デバウンス込み）

currentState が 0（待機中）のとき：
- LCDに「PRESS BUTTON」など待機表示を出す
- pressed == true なら initGame() を呼ぶ
- gameStartTime = now を記録する
- currentState = 1（プレイ中）へ遷移する


currentState が 1（プレイ中）のとき：
- updatePlayer(pressed) でプレイヤー位置を更新する
- readSpeedControl() で速度補正値を取得し scrollSpeed を更新する（任意機能）
- now - lastObstacleTime >= scrollSpeed なら
   - updateObstacles() を呼ぶ
   - 必要なら spawnObstacle() を呼ぶ
   - lastObstacleTime = now に更新する
- now - lastDrawTime >= 100 なら drawFrame() を呼び、lastDrawTime = now に更新する
- checkCollision() が true なら currentState = 2（ゲームオーバー）へ遷移する


currentState が 2（ゲームオーバー）のとき：
- updateScore() で最終スコアを計算する（ゲームオーバー遷移時に1回だけ）
- showGameOver() で最終表示を行う
- pressed == true なら currentState = 0（待機中）へ戻る

```

### `readButton()` — ボタン押下をデバウンス付きで判定

**引数：**
- なし

**戻り値：**
- `bool`：押下イベントがあれば `true`

```
【処理の流れ】
1. digitalRead(PIN_BUTTON) を読む（INPUT_PULLUPなのでLOWが押下）
2. 前回状態と比較し、変化があれば時刻を記録する
3. 変化後50ms以上経過していたら確定入力とみなす
4. 押下イベントが成立したときのみ true を返す

【エラーケース】
- 入力が短時間に連続変化：チャタリングとして無効化する
```

- なし

```
【処理の流れ】
1. pressed が true のときだけ処理する
2. playerLane を 0 ↔ 1 にトグルする
3. 画面外値にならないことを確認する

【エラーケース】
- playerLane が 0/1 以外：0 に戻して継続する
```

- なし

```
【処理の流れ】
1. 乱数でレーン（0 or 1）を決定する
2. 生成位置（画面上端）に障害物を配置する
3. 連続で同一パターンになりすぎないよう必要なら補正する

【エラーケース】
- 障害物枠が満杯：最古の障害物を上書きするか生成をスキップする
```

### `updateObstacles()` — 障害物のスクロール更新

**引数：**
- なし

**戻り値：**
- なし

```
【処理の流れ】
1. すべての障害物を1ステップ下方向へ移動する
2. 画面外へ出た障害物を削除する
3. 生成条件を満たしたら spawnObstacle() を呼ぶ

【エラーケース】
- 配列インデックス異常：更新を中断して安全側（削除）に倒す
```

---

### `checkCollision()` — 衝突判定

**引数：**
- なし

**戻り値：**
- `bool`：衝突したら `true`

```
【処理の流れ】
1. プレイヤー座標（レーン・行）を取得する
2. 各障害物の座標と比較する
3. 同じレーンかつ同じ行なら true を返す
4. どれとも一致しなければ false を返す

【エラーケース】
- 座標値が範囲外：衝突なしとして扱い、次フレームで補正する
```

---

### `updateScore()` — 最終スコア計算

**引数：**
- なし

**戻り値：**
- なし

```
【処理の流れ】
1. 生存時間 = (現在時刻 - gameStartTime) を秒に変換する
2. 倍率 = 1 + floor(生存時間 / 30秒) を計算する
3. score = 生存時間(秒) × 倍率 で最終スコアを算出する
4. 同一ゲーム内で重複計算しないようフラグ管理する

【エラーケース】
- gameStartTime が未設定：score = 0 として処理を終了する
```

---

### `drawFrame()` — LCD描画更新

**引数：**
- なし

**戻り値：**
- なし

```
【処理の流れ】
1. LCDの表示内容をクリアする
2. プレイヤー位置を描画する
3. 障害物を描画する
4. 必要ならスコアや速度情報を描画する
5. 表示ちらつきを抑えるため更新周期は100ms程度に制限する

【エラーケース】
- LCD通信失敗：描画をスキップして次周期で再試行する
```

---

### `showGameOver()` — ゲームオーバー表示

**引数：**
- なし

**戻り値：**
- なし

```
【処理の流れ】
1. LCDの1段目（1行目）に「GAME OVER」を表示する
2. LCDの2段目（2行目）に「SCORE: <最終スコア>」を表示する
3. readButton() でボタン押下を監視する
4. ボタンが押されたら currentState = 0（待機中）に戻す
5. 待機画面表示（例: 「PRESS BUTTON」）に切り替える

【エラーケース】
- スコア値が異常：0として表示する
```

---

### `readSpeedControl()` — 速度補正値の取得（時間があれば実装する）

**引数：**
- なし

**戻り値：**
- `int`：スクロール間隔への補正値（ms）

```
【処理の流れ】
1. analogRead(PIN_POT) で potValue を取得する
2. potValue(0〜1023) を補正値（例: 0〜1300ms短縮）へ map する
3. scrollSpeed = 基本値 - 補正値 で計算し、最小値（例: 700ms）を下回らないよう制限する
4. 補正値または更新後scrollSpeedを返す

【エラーケース】
- potValue が異常値：補正値0としてゲーム継続する
```

## 3. 重要ロジックの設計（必要な場合）

> ※ 複雑なロジック（ゲーム判定、センサー計算、LED点灯パターンなど）は  
> ここでフローチャートや疑似コードで詳しく設計しておきます。

### 3-1. 衝突判定ロジック（`checkCollision()`）

```
【入力】：
- playerLane（0 or 1）
- playerRow（プレイヤー表示行。通常は下段固定）
- obstaclePos[]（各障害物のレーン・行）

【処理】：
1. playerLane が 0/1 以外なら 0 に補正する
2. obstaclePos[] を先頭から順に走査する
3. 各障害物について「同じレーン かつ 同じ行」かを判定する
4. 1つでも一致したら衝突フラグ = true にして探索を終了する
5. 一致がなければ衝突フラグ = false のまま終了する

【出力】：
- 衝突あり: true
- 衝突なし: false
```

### 3-2. 障害物スクロール更新ロジック（`updateObstacles()`）

```
【入力】：
- now（現在時刻）
- lastObstacleTime（前回更新時刻）
- scrollSpeed（更新周期ms）
- obstaclePos[]（障害物配列）

【処理】：
1. now - lastObstacleTime < scrollSpeed の間は何もしない（早すぎる更新を防止）
2. 更新条件を満たしたら、全障害物の行を1つ下へ進める
3. 画面外（最下段より下）に出た障害物を削除する
4. 空いた枠があれば spawnObstacle() の生成判定へ渡す
5. lastObstacleTime = now に更新する

【出力】：
- 更新済み obstaclePos[]
- 更新済み lastObstacleTime
```

### 3-3. 障害物生成ルール（`spawnObstacle()`）

```
【入力】：
- obstaclePos[] の使用状況
- 乱数値（レーン決定用）

【処理】：
1. 新規生成位置を「最上段」に固定する
2. レーンは 0/1 の乱数で選ぶ
3. 直前フレームと同じパターンが連続しすぎる場合は再抽選する
4. 配列が満杯なら「生成スキップ」または「最古要素上書き」のどちらかに統一する
5. 生成後の障害物が即衝突しないこと（同フレーム同座標）を確認する

【出力】：
- 追加後 obstaclePos[]（または変更なし）
```

### 3-4. プレイヤー移動処理（`updatePlayer()`）

```
【入力】：
- pressed（ボタン押下イベント）
- playerLane（現在のレーン。0:左, 1:右）

【処理】：
1. pressed が false なら何もせず終了する
2. pressed が true のときだけ playerLane を 0 ↔ 1 で切り替える
3. 切り替え後の playerLane が 0/1 以外なら 0 に補正する
4. 切り替え結果を次フレーム描画に反映する

【出力】：
- 更新後 playerLane（0 または 1）
```

## 5. AIレビュー記録（詳細設計版）

### Q1: 実装上の問題確認
> 「この詳細設計書に書いた関数と処理フローをもとにArduinoコードを書きます。  
> バグになりやすい箇所・処理の抜け・型の問題はありますか？」

**AIの回答（要約）：**
- グローバル変数ブロックでコードフェンスが途中で閉じており、設計書から実装へ転記する際に変数の貼り漏れ・型漏れが起きやすい。
- `showGameOver()` と `loop()` の両方でボタン押下監視を持つと責務が重複し、状態遷移の二重実行リスクがある。
- `updateScore()` は「1回だけ計算」の意図に対して、重複計算防止フラグが未定義だと毎ループ再計算される可能性がある。
- `LiquidCrystal_I2C` は実装差で初期化API（`begin`/`init`）が異なるため、利用ライブラリを先に確定した方がよい。
- 障害物配列の設計は2次元化で改善したが、座標の意味（行/列）と空状態の定義を固定しないと判定バグが出やすい。

**対応した内容：**
- `buttonPressed` の重複定義を解消した。
- `obstaclePos` を `int obstaclePos[2][16]` に見直し、障害物管理の拡張性を確保した。
- `showGameOver()` の仕様を「1行目: GAME OVER / 2行目: SCORE表示 / 押下で待機へ遷移」に統一した。
- 重要ロジック（衝突判定・スクロール更新・生成ルール・プレイヤー移動）を分離して、処理順序を明文化した。
- 次の実装時に、状態管理は `loop()` 側へ集約し、`showGameOver()` は表示専用にする方針を確認した。

### Q2: テスト仕様の確認
> 「このテスト仕様書で、必須機能がすべて検証できていますか？  
> テストが不足している項目や、境界値テストが必要な箇所を教えてください。」
