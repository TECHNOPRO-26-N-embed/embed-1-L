## 1. システム全体像

### 1-1. ブロック図（入力 → 処理 → 出力）

> ※ Arduinoを中心に、「何が入力で何が出力か」を整理します。
> ASCII図または別紙の写真を貼り付けてください。

**記入例：**
```
【入力】               【Arduino UNO R3】          【出力】
超音波センサー ────→┐                       ┌──→ LED（赤・緑）
タクトスイッチ ────→┤    状態管理・制御      ├──→ パッシブブザー
                    └─────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】               【Arduino UNO R3】          【出力】
タクトスイッチ ────→┐                       ┌──→ LCDモニター（ゲーム表示）
ポテンショメーター ─→┤  状態管理・当たり判定   ├──→ （できれば）パッシブブザー
                    └  スコア計算・描画制御 ┘
```

### 1-2. 状態遷移図（システムの「ふるまい」）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> 状態（○）と、状態が切り替わる条件（矢印の横）を書いてください。

**記入例：**
```
[電源ON]
  ↓ 初期化完了
[待機中] ─── センサー検知 ───→ [動作中] ─── 終了条件 ───→ [完了]
  ↑                                 │
  └──────── リセット ────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON]
  ↓ 初期化完了
[待機中] ─── ボタン押下 ───→ [プレイ中] ─── 衝突 ───→ [ゲームオーバー]
   ↑                                │                     │
   └────────────── 再起動/リセット ───┴── ボタン押下 ────┘
```

**状態の意味を説明してください：**

| 状態名 | この状態のとき、何をしているか |
|:--|:--|
| 初期化 | LCD初期化、ピンモード設定、変数初期化を行う |
| 待機中 | スタート待ち表示を出し、ボタン入力を監視する |
| プレイ中 | ボタン入力でプレイヤー位置更新、障害物生成/移動、衝突判定、スコア更新を行う |
| ゲームオーバー | 最終スコア表示、全状態をリセットし再スタート入力を待つ |

## 2. ハードウェア設計

### 2-1. 部品一覧と接続ピン

> ※ 使う部品とArduinoのどのピンに繋ぐかを決めます。
> デジタル（D2〜D13）/ アナログ（A0〜A5）/ 通信（I2C: A4/A5, SPI: D10〜D13）

| 部品名 | 型番・種類 | Arduinoピン | 入力/出力 | 備考 |
|:--|:--|:--|:--|:--|
| タクトスイッチ | Button | D2 | 入力 | INPUT_PULLUP使用（押下でLOW） |
| LCDモニター | LCD1602 Module | A4(SDA), A5(SCL) | 出力 | 表示用。アドレスは0x27想定 |
| ポテンショメーター | Potentiometer | A0 | 入力 | 任意機能。速度調整に使用 |
| パッシブブザー | Passive Buzzer | D8 | 出力 | 任意機能。開始/終了ジングル | 

> [!TIP]
> **ピン競合チェック：** I2C（A4/A5）・SPI（D10〜D13）は複数部品で共用します。
> デジタルピン（D2〜D9）は原則1部品1ピンです。D0/D1はシリアル通信に使われるため避けること。

### 2-2. 使用する抵抗・回路メモ

> ※ 抵抗値の計算や、注意が必要な接続を記録してください。

| 用途 | 計算・メモ |
|:--|:--|
| ボタン入力安定化 | 内部プルアップ（INPUT_PULLUP）を使うため外付け抵抗なし |
| ブザー回路 | パッシブブザーをD8へ直結（音が大きかったりしたら抵抗を挟む） |
| その他 | LCDはI2C接続のため信号線はSDA/SCLのみ。GND共通を徹底する |

### 3-2. 関数一覧（役割の定義）

> ※ プログラムを小さな「処理の部品（関数）」に分割します。
> **ここではコードを書かず、「何をする関数か」だけ日本語で書いてください。**
> 具体的な実装は詳細設計書（detailed_design.md）で行います。

| 関数名 | 何をする関数か（1〜2行で） | 引数（入力） | 戻り値（出力） |
|:--|:--|:--|:--|
| `setup()` | Arduinoの初期化・ピンモード設定 | なし | なし |
| `loop()` | 状態に応じて各関数を呼び出すメインループ | なし | なし |
| `initGame()` | プレイヤー位置、障害物配列、スコア、速度を初期値に戻す | なし | なし |
| `readButton()` | ボタンの押下をデバウンス付きで判定する | なし | bool |
| `updatePlayer()` | ボタン入力に応じてプレイヤーの左右レーンを切り替える | bool pressed | なし |
| `spawnObstacle()` | ルール付き乱数で新しい障害物を生成する | なし | なし |
| `updateObstacles()` | 障害物を上段から下段へスクロール移動させる | なし | なし |
| `checkCollision()` | プレイヤー位置と障害物位置の一致を判定する | なし | bool |
| `updateScore()` | ゲームオーバー時に生存時間からスコアを一括計算する（30秒ごとに倍率+1） | なし | なし |
| `drawFrame()` | LCDにプレイヤーと障害物、必要情報を描画する | なし | なし |
| `showGameOver()` | ゲームオーバー表示と最終スコア表示を行う | なし | なし |
| `readSpeedControl()` | ポテンショメーター値から速度補正量を取得する（任意機能） | なし | int |

### 3-3. タイミング設計（delay vs millis）

> ※ 複数の処理を同時に動かす場合、`delay()`を使うと他の処理が止まります。
> 各タイマーが `delay` か `millis` かを事前に決めておきます。

| 処理 | 周期 | delay / millis | 理由 |
|:--|:--|:--|:--|
| ボタン入力監視 | 10ms | **millis** | 応答時間0.1〜0.5秒を満たすため |
| 障害物スクロール | 2000ms〜700ms | **millis** | ゲーム進行と入力監視を同時に行うため |
| LCD再描画 | 100ms | **millis** | ちらつきを抑えつつ表示更新するため |
| スコア計算（ゲームオーバー時） | ゲームオーバー時のみ | 不要（イベント時実行） | リアルタイム更新をやめ、終了時に一回だけ計算するため |
| （できれば）ブザー再生 | 3000ms以内 | **millis** | 音再生中もボタン監視を止めないため |

## 4. 異常系・エラー処理の方針

> ※ 要件定義書の「非機能要件（3-4）」を具体化します。
> 詳細な実装方法は詳細設計書で書きます。ここでは「どう検知し、どう対応するか」の方針だけ。

| 異常ケース | 検知方法 | 対応方針 |
|:--|:--|:--|
| ボタン入力が不安定 | 前回確定入力から50ms未満の変化を検出 | チャタリングとみなし無効化する |
| ボタンが長時間反応しない | 一定時間（例:10秒）状態変化なし | 画面に「INPUT ERROR」を表示し停止する |
| ポテンショメーター値が異常 | analogRead値が0/1023付近で固定 | 速度補正を0としてゲーム継続する |
| 処理遅延が発生 | ループ周期が目標を超過（例:500ms超） | 描画更新頻度を落として入力監視を優先する |

## 6. AIレビュー記録（基本設計版）

### Q1: 構造の妥当性確認
> 「この基本設計書をもとに Arduino UNO R3 でコードを書くとき、
> ピン不足・ライブラリの競合・状態設計の漏れはありますか？」

**AIの回答（要約）：**
- I2C接続のLCDを使う構成ならピン不足は発生しにくい。
- `delay()`中心で実装すると入力取りこぼしが起きやすいため、状態管理とタイミング管理は`millis()`で統一した方がよい。
- 状態は「待機中 / プレイ中 / ゲームオーバー」の3つに絞ると漏れが少ない。

**対応した内容：**
- ピン割り当てをD2/A0/A4/A5/D8で固定し、競合しないよう設計した。
- タイミング設計をすべて`millis()`ベースに統一した。
- 状態遷移図を3状態中心に修正し、再スタート条件を明記した。

---

### Q2: 詳細設計への引き継ぎ確認
> 「この基本設計書から詳細設計に進むとき、
> まだ決まっていない・詳細化が必要な部分はどこですか？」

**AIの回答（要約）：**
- 詳細設計では、障害物データ構造（配列サイズ、移動タイミング、削除条件）を数値で確定する必要がある。
- デバウンス時間、初期スクロール速度、最速速度、スコア倍率などの定数を明文化する必要がある。
- 任意機能（音・速度可変）は基本機能と分離し、後付けできる形で設計すべき。

**対応した内容：**
- 関数一覧に`spawnObstacle()`、`updateObstacles()`、`readSpeedControl()`を追加し、詳細化ポイントを明確化した。
- 実装計画で任意機能を最後に配置し、段階実装の順序を固定した。
- 異常系方針に入力異常・処理遅延時の対応を追加した。
