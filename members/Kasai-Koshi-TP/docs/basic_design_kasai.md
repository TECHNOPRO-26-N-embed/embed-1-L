## 1. システム全体像

### 1-1. ブロック図（入力 → 処理 → 出力）

> ※ Arduinoを中心に、「何が入力で何が出力か」を整理します。  
> ASCII図または別紙の写真を貼り付けてください。

**記入例：**
```
【入力】               【Arduino UNO R3】          【出力】
超音波センサー ────→┐                       ┌──→ LED（赤・緑）
タクトスイッチ ────→┤    状態管理・制御      ├──→ パッシブブザー
                    └─────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】             【Arduino UNO R3】           【出力】
ボタン ────────→┐                        ┌──→ ブザー（曲再生）
                 │   状態管理・制御       ├──→ LEDディスプレイ（歌詞表示）
                 └────────────────────┘


```

### 1-2. 状態遷移図（システムの「ふるまい」）

> ※ ガジェットが「どんな状態を持つか」を定義します。  
> 状態（○）と、状態が切り替わる条件（矢印の横）を書いてください。

**記入例：**
```
[電源ON]
  ↓ 初期化完了
[待機中] ─── センサー検知 ───→ [動作中] ─── 終了条件 ───→ [完了]
  ↑                                 │
  └──────── リセット ────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON]
  ↓
[ボタン待機] ── ボタン検知 ──→ [曲再生・歌詞表示] ── 曲終了 ──→ [終了]
     ↑                                             　　　　　　  │
     └────────────── リセット ──────────────────────────────────┘
・「曲再生・歌詞表示」状態でボタンが押された場合は、入力を無視し状態は変化しない。

```

**状態の意味を説明してください：**

| 状態名 | この状態のとき、何をしているか |
|:--|:--|
| 待機中 | ボタンの入力を監視しながら、LEDを一定間隔で点滅 |
| ボタン待機  | ボタンを押すまで待機、LEDなどは点滅しないそうてい |
| ボタン検知 | ボタンが押されたら、次の処理を実行 |
| 曲再生・歌詞表示 | 曲を流し、歌詞を表示（歌詞は都度表示が変わる） |
| 曲終了 | 曲が終了しすべての機能を停止 |
| 終了 | 終了し待機に戻る |

## 3. ソフトウェア構造設計

### 3-1. 使用するライブラリ

> ※ 使うArduinoライブラリを調べておきます。

| ライブラリ名 | 用途 | インストール方法 |
|:--|:--|:--|
| LiquidCrystal_I2C | LCDディスプレイの制御 | ライブラリマネージャで検索 |
| Wire | I2C通信の土台（LCDで内部的に使用） | 標準ライブラリ（インストール不要） |
| なし(tone関数) | passive buzzerで音を鳴らす | 標準関数なので追加不要 |

### 3-2. 関数一覧（役割の定義）

> ※ プログラムを小さな「処理の部品（関数）」に分割します。  
> **ここではコードを書かず、「何をする関数か」だけ日本語で書いてください。**  
> 具体的な実装は詳細設計書（detailed_design.md）で行います。

| 関数名 | 何をする関数か（1〜2行で） | 引数（入力） | 戻り値（出力） |
|:--|:--|:--|:--|
| `setup()` | Arduinoの初期化・ピンモード設定・LCDの初期化 | なし | なし |
| `loop()` | 状態に応じて各関数を呼び出すメインループ | なし | なし |
| `readButton()` | ボタン状態を読みより、押下判定を行う | なし | あり(bool型) |
| `start` | 再生開始の初期化 | なし | なし |
| `updateLyrics` | 経過時間に応じて表示する歌詞を切り替える | あり(unisigned long型) | なし |
| `playMelody` | 経過時間に応じてブザーの音程・長さを制御する | あり(unisigned long型) | なし |
| `Finish` | 曲終了の判定を行う | あり(unisigned long型) | あり(boo型) |
| `stop` | 音停止・LCDクリアなど終了処理を行い状態へ戻す | なし | なし |

### 3-3. タイミング設計（delay vs millis）

> ※ 複数の処理を同時に動かす場合、`delay()`を使うと他の処理が止まります。  
> 各タイマーが `delay` か `millis` かを事前に決めておきます。

| 処理 | 周期 | delay / millis | 理由 |
|:--|:--|:--|:--|
| ボタン監視 | 1s | mills | 連打の対策のため常時監視が必要 |
| 歌詞表示切替 | 歌詞ごとに個別設定（歌詞と表示時間を配列に格納する） | mills | 歌詞表示と曲再生が一緒に行うため |
| メロディ再生 | 一音ずつ個別設定（音の長さを配列で管理し調整予定） | mills | 曲の進行とほか処理を同時に行うため |

## 4. 異常系・エラー処理の方針

> ※ 要件定義書の「非機能要件（3-4）」を具体化します。  
> 詳細な実装方法は詳細設計書で書きます。ここでは「どう検知し、どう対応するか」の方針だけ。

| 異常ケース | 検知方法 | 対応方針 |
|:--|:--|:--|
| センサー値が範囲外 | 例：value < 0 または value > 1023 | 例：その読み値を無視してスキップ |
| チャタリング（ボタン） | millsで入力時間を管理し前回入力から50ms未満の入力を無効 | デバウンス処理を入れる |
| 歌詞データの欠落 | 例：表示すべき歌詞が配列概算賞などで見つからない | 空白表示またはエラー表示（LEDディスプレイ） |
| ブザー出力エラー | 例:tone関数で異常な周波数を指定 | 異常値の場合は音を鳴らさずスキップ |
| ボタンの連打 | 例：曲再生中にボタンが複数回押された | 曲再生中は追加の入力を無視 |

## 6. AIレビュー記録（基本設計版）

### Q1: 構造の妥当性確認
> 「この基本設計書をもとに Arduino UNO R3 でコードを書くとき、  
> ピン不足・ライブラリの競合・状態設計の漏れはありますか？」

**AIの回答（要約）：**
ピン不足：
　LCD（I2C接続）、ボタン、ブザーのみなのでピン数は十分です。I2CはA4/A5を共用するため他のデジタルピンと競合しません。

ライブラリの競合：
　LiquidCrystal_I2CとWireは問題なく共存できます。tone関数も標準で競合なしです。

状態設計の漏れ：
　「待機」「再生・歌詞表示」「終了」など主要な状態は網羅されています。
　ただし、エラー時（歌詞データ欠落やボタン連打時など）の遷移や、再生中のボタン入力無視なども明記しておくとより堅牢です。

**対応した内容：**
エラー時について追加
